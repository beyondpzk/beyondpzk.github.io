# 原生多模：从 Janus 到 Janus-Pro，一个 Transformer 同时理解与生成

> **论文**：*Janus: Decoupling Visual Encoding for Unified Multimodal Understanding and Generation*  
> **论文**：*Janus-Pro: Unified Multimodal Understanding and Generation with Data and Model Scaling*

## 摘要

「原生多模态」（Native Multimodal）正在取代早期「拼接式」多模态方案，成为视觉-语言模型的新范式。与分别训练理解编码器和生成解码器、再用 VLM 做后处理的系统不同，原生多模态主张**用同一个自回归 Transformer 同时完成视觉理解、文本推理和图像生成**。

DeepSeek 的 **Janus** 与 **Janus-Pro** 是这一路线最具代表性的开源工作：

- 提出**解耦视觉编码**（Decoupled Visual Encoding），用独立的编码器分别处理理解任务的高维语义特征和生成任务的细粒度像素特征，从根本上缓解「理解-生成粒度冲突」。
- 采用单一 LLM 主干，所有模态共享同一个 next-token 预测目标。
- Janus-Pro 在数据规模、训练策略和模型尺寸上进一步扩展，7B 版本在 MMBench、SeedBench 等理解基准与 GenEval、DPG-Bench 等生成基准上同时达到同尺寸开源 SOTA。

本文从架构动机、模型设计、三阶段训练、数据 scaling、评测结果到局限与未来方向，系统梳理 Janus 系列的技术报告。

## 一、背景：为什么需要「原生多模态」？

### 1.1 传统方案的三条路线

多模态大模型在 2023-2024 年主要分化为三类：

| 路线 | 代表 | 特点 | 局限 |
|---|---|---|---|
| **理解型 VLM** | CLIP + LLaVA、Qwen-VL、InternVL | 用视觉编码器抽取语义特征，对齐到 LLM | 只能理解，不能生成图像 |
| **生成型文生图/视频模型** | Stable Diffusion、SDXL、Sora | 用扩散模型或自回归模型生成像素 | 只能生成，缺乏复杂推理 |
| **拼接式统一模型** | Show-o、SEED-X、Emu3 | 一个框架内挂载理解模块和生成模块 | 模块间目标不一致，训练复杂 |

理解任务关注**高层语义**（类别、关系、OCR、推理），生成任务关注**低层纹理与结构**（颜色、边缘、空间布局）。二者对视觉表征的需求天然冲突：

- 理解希望特征对纹理噪声鲁棒、语义抽象；
- 生成希望保留细粒度空间细节，以便精确重建像素。

传统统一模型若复用同一个视觉编码器，往往两头不讨好。

### 1.2 原生多模态的核心主张

原生多模态（Native Multimodal）提出：**不要为不同任务单独设计模块，而是让模型在预训练阶段就原生地学会跨模态统一表示**。关键特征包括：

1. **统一架构**：单一 Transformer 处理文本、图像 token；
2. **统一目标**：所有任务都用 next-token prediction；
3. **统一输入输出**：图像被离散化为 token，与文本 token 一起进入自回归模型；
4. **端到端训练**：理解和生成数据混合训练，而不是分阶段拼模块。

Janus 是原生多模态路线的早期开源代表，Janus-Pro 则进一步验证了这一路线的可扩展性。

## 二、Janus：解耦视觉编码的统一框架

### 2.1 核心思想：Decoupled Visual Encoding

Janus 的关键洞察是：**理解路径和生成路径需要不同的视觉编码器，但可以在同一个 LLM 内部统一**。

- **理解编码器（Understanding Encoder）**：使用预训练的 SigLIP 等语义视觉编码器，输出高维、语义化的视觉特征，适合 VQA、OCR、推理等任务。
- **生成编码器（Generation Encoder）**：使用 VQ-VAE / VQ Tokenizer 等像素级 tokenizer，输出离散的图像 token，适合重建和生成细节。
- **统一 LLM 主干**：理解和生成的 token 都映射到同一个语言模型输入空间，由同一个自回归 Transformer 处理。

这与「拼接式」方案有本质区别：拼接式通常把理解 VLM 和生成 diffusion 当两个独立子系统，而 Janus 的 LLM 同时承担理解与生成的大脑。

### 2.2 模型架构

Janus 的整体架构包含五个部分：

1. **LLM 主干**：基于 DeepSeek-LLM 1.3B，作为统一的自回归推理核心。
2. **理解视觉编码器**：SigLIP-Large-Patch16-384，用于提取图像语义特征。
3. **理解适配器（Understanding Adapter）**：两层 MLP，将 SigLIP 特征投影到 LLM 输入空间。
4. **生成视觉编码器**：VQ tokenizer，把图像编码为离散 token（下采样 16×，codebook 大小 16384）。
5. **生成适配器（Generation Adapter）**：两层 MLP，将 VQ token 嵌入投影到 LLM 输入空间；另外有一个随机初始化的 **image generation head**，用于把 LLM 输出映射回 VQ codebook 预测图像 token。

输入序列示例：

```text
[text tokens] <image> [visual tokens] </image> [text tokens]
```

其中 `<image>` 和 `</image>` 是特殊标记。理解任务中 visual tokens 来自 SigLIP + adaptor；生成任务中，输入文本作为 prompt，模型自回归地输出 VQ image tokens，再通过 VQ decoder 还原为图像。

### 2.3 训练策略：三阶段统一训练

Janus 采用三阶段训练流程，逐步解锁参数：

#### Stage I：视觉-语言对齐预训练

- **目标**：让 LLM 学会理解视觉语义，同时学会把文本映射到图像生成空间。
- **可训练参数**：理解适配器、生成适配器、image generation head。
- **冻结参数**：理解视觉编码器、生成视觉编码器（VQ tokenizer）、LLM 主干。
- **数据**：
  - 理解数据：ShareGPT4V、ALLaVA 等图文对，约 125 万样本；
  - 生成数据：ImageNet 类别名→图像（约 120 万），用于让模型学习基础视觉生成。
- **训练方式**：所有样本统一用 next-token prediction，视觉 token 也参与 loss。

#### Stage II：统一预训练

- **目标**：扩展数据规模，让 LLM 在真实多任务数据上统一优化。
- **可训练参数**：除理解编码器和生成编码器外，其余全部打开（LLM、两个适配器、image head）。
- **数据**：
  - 多模态理解数据：约 4800 万样本；
  - 纯文本数据：约 9000 亿 token（保持语言能力）；
  - 文生图数据：约 1200 万样本。
- **数据配比**：理解 : 文本 : 生成 ≈ 2 : 3 : 5。

#### Stage III：监督微调（SFT）

- **目标**：提升指令遵循能力和生成质量。
- **可训练参数**：全部打开，包括理解视觉编码器（SigLIP）。
- **数据**：
  - 多模态理解指令数据；
  - 文生图指令数据；
  - 文本指令数据。
- **数据配比**：理解 : 文本 : 生成 ≈ 5 : 1 : 4。

### 2.4 关键设计决策

- **为什么解耦视觉编码？**
  论文通过实验发现，如果用同一个视觉编码器同时服务理解和生成，理解性能会显著下降。SigLIP 的 patch-level 特征语义强但缺乏像素细节，VQ tokenizer 的 token 适合生成但包含太多低层噪声，不适合高层推理。解耦后两条路径各司其职。

- **为什么 Stage III 才解锁 understanding encoder？**
  早期冻结 SigLIP 是为了保留其强大的语义先验；Stage III 用高质量指令数据微调，使其更贴合对话和复杂推理任务。

- **为什么 image generation head 随机初始化？**
  因为 VQ codebook 的预测空间与文本词表差异大，不能复用 LLM 的 LM head，需要单独学习。

## 三、Janus-Pro：数据与模型 Scaling

Janus-Pro 在 Janus 的基础上做了三方面的扩展：**训练策略优化、数据 scaling、模型尺寸扩展**。

### 3.1 训练策略优化

Janus-Pro 对 Janus 的三阶段训练做了如下调整：

| 阶段 | Janus | Janus-Pro 改进 |
|---|---|---|
| Stage I | 用 ImageNet 类别名生成图像 | 去掉 ImageNet，直接用真实文生图数据，训练更长 |
| Stage II | 理解:文本:生成 = 2:3:5 | 增加理解和生成数据量，优化配比 |
| Stage III | 理解:文本:生成 = 5:1:4 | 加入更多多模态理解指令数据，强化指令遵循 |

具体改进包括：

1. **Stage I 训练更长**：让适配器和 image head 在更丰富的真实文本-图像对上充分收敛。
2. **Stage I 不再用 ImageNet**：ImageNet 的类别名过于简短，不利于学习复杂的文本-图像对齐。Janus-Pro 直接用高质量文生图数据（如 DPG、MJ-style captions）。
3. **Stage II 数据扩展**：理解数据增加约 **9000 万样本**；生成数据引入约 **7200 万合成美学数据**，真实数据与合成数据比例约 1:1。
4. **Stage III 数据配比调整**：进一步增加多模态理解指令数据，改善 VQA、OCR、推理表现。

### 3.2 数据 scaling 细节

Janus-Pro 的数据策略体现了原生多模态训练的核心：**理解和生成数据需要按一定比例混合，以避免模型偏向单一任务**。

- **理解数据**：包含通用图文对、OCR、图表、科学图像、细粒度视觉定位等，覆盖多种分辨率和场景。
- **生成数据**：
  - 真实数据：高质量文生图数据集；
  - 合成数据：使用 DeepSeek 自有的美学评分模型过滤和增强，提升生成图像的美感和指令遵循度。
- **文本数据**：持续预训练文本语料，防止多模态训练导致语言能力退化。

### 3.3 模型配置

Janus-Pro 提供两个尺寸：

| 配置 | Janus-Pro-1B | Janus-Pro-7B |
|---|---|---|
| LLM 参数 | 1.5B | 7B |
| LLM 层数 | 24 | 30 |
| Hidden size | 2048 | 4096 |
| Attention heads | 16 | 32 |
| Context window | 4096 | 4096 |
| 理解编码器 | SigLIP-Large-Patch16-384 | SigLIP-Large-Patch16-384 |
| 生成编码器 | VQ tokenizer（16× downsample, 16384 codebook） | VQ tokenizer |
| 可训练新增参数 | 约 70M（adaptor + head） | 约 170M |

注意：Janus-Pro-7B 的 LLM 实际使用 7B 参数，配合约 170M 新增参数；Janus-Pro-1B 类似，但规模更小。

### 3.4 训练超参数

Janus-Pro 使用标准的 AdamW 优化器与 cosine learning rate schedule，关键超参数如下（以论文 Table 1/2 为准）：

| 阶段 | 学习率 | Warmup | Steps | Batch size | 数据比例（理解:文本:生成） |
|---|---|---|---|---|---|
| Stage I | 1e-3（adaptor） | 100 steps | 50K+ | 4096 | 1:0:1 |
| Stage II | 4e-4 | 500 steps | 200K+ | 4096 | 2:3:5（Janus）/ 优化后（Janus-Pro） |
| Stage III | 4e-5 | 200 steps | 20K+ | 256 | 5:1:4 |

所有阶段都使用 mixed-precision 训练，Stage II 和 Stage III 由于数据量大，通常需要多节点并行。

## 四、评测结果

### 4.1 理解能力

Janus-Pro 在主流多模态理解基准上表现如下：

| 基准 | Janus-1.3B | Janus-Pro-1B | Janus-Pro-7B |
|---|---|---|---|
| MMBench-Dev | 69.4 | 72.3 | 79.2 |
| MMBench-Test | 69.2 | 72.0 | 78.4 |
| SEEDBench-Image | 68.7 | 71.1 | 77.7 |
| POPE | 86.2 | 87.3 | 89.3 |
| MME-Perception | 1416.8 | 1502.4 | 1634.2 |
| GQA | 59.8 | 61.7 | 65.9 |
| MMMU | 31.5 | 34.1 | 40.1 |
| MM-Vet | 35.9 | 39.2 | 51.1 |

可以看到，Janus-Pro-7B 在理解任务上超过了 Qwen2-VL-7B、InternVL2-8B 等同尺寸理解型 VLM，验证了原生多模态路线在理解任务上的竞争力。

### 4.2 生成能力

在文本到图像生成基准上：

| 基准 | Janus | Janus-Pro-1B | Janus-Pro-7B |
|---|---|---|---|
| GenEval Overall | 0.45 | 0.52 | 0.58 |
| DPG-Bench Overall | 64.6 | 69.4 | 74.2 |

Janus-Pro 在生成任务上显著优于 Janus，并在 GenEval 和 DPG-Bench 上达到同尺寸开源模型中的先进水平。

### 4.3 统一能力对比

Janus-Pro 的关键优势在于**同时覆盖理解和生成**，而传统理解型 VLM 无法生成图像，传统生成模型无法理解图像。在以下统一能力上，Janus-Pro 表现突出：

- **图像理解 + 图像生成**：给定一张图，模型能描述内容，并根据描述生成新图。
- **多轮图文对话**：支持图像输入、文本提问、图像输出的混合多轮交互。
- **视觉推理 + 创意生成**：例如根据图表数据生成可视化图像，或根据草图生成艺术化图像。

## 五、关键问题讨论

### 5.1 Janus 是「以 next token 方式输出动作」吗？

这里需要区分：Janus 是 VLM，不是 VLA，因此不存在「动作输出」。但在生成图像时，Janus 确实是**以 next-token prediction 的方式自回归地输出离散图像 token**，然后由 VQ decoder 解码为像素。每一个 image token 的预测与文本 token 的预测共享同一个 LM head 之前的表征空间，只是最后通过独立的 image generation head 映射到 codebook。

### 5.2 全参数微调后为什么还能保持语言能力？

Janus/Janus-Pro 在 Stage II 和 Stage III 都混合了**大量纯文本数据**（与理解、生成数据按比例混合）。这样即使 LLM 主干参与训练，文本能力也不会严重退化。此外，Stage III 的文本指令数据有助于维持甚至提升对话和推理能力。

### 5.3 训练时只用图像生成数据，还是也用了 VLM 数据？

Janus-Pro 的训练同时使用了：

- 多模态理解数据（类似 VLM 训练数据）；
- 文本数据（维持 LLM 能力）；
- 文生图数据（维持生成能力）。

三阶段数据配比不同，但始终是多任务混合训练，而不是只训练生成或只训练理解。

### 5.4 解耦编码器 vs 单一编码器

论文中的消融实验显示：

- 若用同一个 SigLIP 编码器服务生成任务，生成质量极差，因为 SigLIP 特征丢失了像素细节。
- 若用同一个 VQ tokenizer 服务理解任务，理解准确率显著下降，因为 VQ token 包含太多低层噪声。
- 解耦编码器在两项任务上都取得了最佳平衡。

## 六、其他原生多模态代表：Gemini 与 Qwen3-VL

Janus/Janus-Pro 是开源界「统一理解与生成」的典型代表，但「原生多模态」这一概念并不局限于此。业界还有两条同样重要的路线：

1. **原生多模态理解**：从预训练阶段就同时学习文本、图像、视频、音频，输出统一语义表征；
2. **原生多模态输入输出**：不仅理解多种模态，还能在同一模型内生成图像/音频等。

下面以 **Google Gemini** 和 **Qwen3-VL** 为例，说明原生多模态的多样性。

### 6.1 Google Gemini：从预训练开始就是多模态

Google DeepMind 在 Gemini 1.0 技术报告中明确强调，Gemini 系列是**从头开始原生多模态训练**（natively multimodal from pretraining）的模型，而不是先训练文本模型再拼接视觉/音频模块。

> *"Gemini models are multimodal from the beginning and ... trained jointly on text, images, audio, and video."* — Gemini 1.0 Technical Report

其核心架构可概括为：

```text
h_unified = Transformer([E_text(x_t); E_vision(x_v); E_audio(x_a)])
```

即使用**模态专属编码器**（文本 tokenizer、视觉编码器、USM 音频编码器）将不同模态投影到同一维度，再输入一个统一的 Transformer decoder。训练目标同时包含各模态任务损失和跨模态对比/对齐损失。

**Gemini 的多模态演进**：

| 版本 | 发布时间 | 原生多模态能力 |
|---|---|---|
| Gemini 1.0 / 1.5 | 2023-2024 | 文本、图像、音频、视频统一理解；长上下文 1M+ tokens |
| Gemini 2.0 Flash | 2024-2025 | 在理解基础上增加**原生图像生成与编辑**（native image generation），可输出图文交错内容 |

Gemini 2.0 Flash 的图像生成不是调用 Imagen 3 等外部扩散模型，而是由同一模型自回归输出图像 token 并解码为图像，这与 Janus-Pro 的生成机制理念一致。不过 Gemini 的架构细节并未完全开源。

### 6.2 Qwen3-VL：开源原生多模态理解的标杆

阿里巴巴通义的 **Qwen3-VL**（arXiv:2511.21631）是开源社区中最强的原生多模态理解模型之一。其技术报告摘要明确指出：

> *"Qwen3-VL natively supports interleaved contexts of up to 256K tokens, seamlessly integrating text, images, and video."*

**关键架构特点**：

- **统一 Transformer 主干**：基于 Qwen3 LLM，文本、图像、视频 token 在同一自回归框架下处理。
- **增强型 Interleaved-MRoPE**：把 2D 空间位置与视频时序位置统一编码，提升图像和视频的空间-时序推理。
- **DeepStack**：融合 ViT 多层特征，增强细粒度视觉-语言对齐。
- **文本化时间对齐**：对视频采用显式时间戳文本对齐，替代传统 T-RoPE，提升长视频事件定位精度。
- **模型规模**：Dense 版本 2B/4B/8B/32B，MoE 版本 30B-A3B / 235B-A22B。

**需要注意的区分**：

- Qwen3-VL 是**原生多模态理解模型**，它统一处理文本、图像、视频输入并生成文本输出；
- 但 Qwen3-VL **本身不生成图像**；阿里的图像生成由独立模型 **Qwen-Image / Qwen-Image-Edit** 负责。
- 因此 Qwen3-VL 属于「原生多模态理解」，而 Janus-Pro 和 Gemini 2.0 Flash 属于「原生多模态统一理解与生成」。

### 6.3 三类原生多模态路线对比

| 路线 | 代表 | 输入模态 | 输出模态 | 核心特征 |
|---|---|---|---|---|
| **原生多模态理解** | Gemini 1.5、Qwen3-VL、GPT-4V、InternVL3 | 文本/图像/视频/音频 | 文本 | 预训练阶段即联合学习多模态表征，统一 Transformer 处理 |
| **原生多模态统一理解+生成** | Janus-Pro、Gemini 2.0 Flash、GPT-4o | 文本/图像/视频/音频 | 文本/图像/音频 | 同一模型既能理解又能生成，图像也作为 token 自回归输出 |
| **原生多模态 embedding/检索** | Gemini Embedding 2、Qwen3-VL-Embedding | 文本/图像/视频/音频/文档 | embedding 向量 | 把多种模态映射到统一向量空间，服务 RAG 与检索 |

这三条路并非互斥，而是原生多模态在不同应用场景下的延伸。Janus-Pro 的独特之处在于**用完全开源的方式证明了「统一理解与生成」的可行性**，而 Gemini 和 Qwen3-VL 则分别在闭源商业系统和开源理解系统上推动了原生多模态的普及。

## 七、局限与未来方向

### 7.1 当前局限

1. **生成质量仍落后于顶尖扩散模型**：在照片级真实感和高分辨率生成上，Janus-Pro 与 FLUX、SD3、DALL-E 3 等扩散模型仍有差距。
2. **图像分辨率受限**：当前主要支持 384×384 的理解输入和 384×384 的生成输出，高分辨率扩展仍是挑战。
3. **生成速度较慢**：自回归生成需要逐个 token 采样，比单步/少步扩散模型慢。
4. **视频与音频未统一**：Janus 系列目前主要覆盖图像-文本，尚未原生扩展到视频、音频等更多模态。

### 7.2 未来方向

1. **Scaling 到更大模型**：验证 13B、70B 规模下原生多模态的统一能力边界。
2. **更高效的视觉 tokenizer**：开发同时适合理解和生成的统一视觉编码器，或更高效的离散 tokenizer。
3. **多模态 in-context learning**：让模型通过少量示例同时学习新任务的理解和生成。
4. **与 VLA 结合**：将原生多模态模型作为 VLA 的感知-推理核心，统一处理视觉、语言、动作。

---

## 八、总结

DeepSeek Janus / Janus-Pro 为「原生多模态」提供了坚实的开源范例：

- **架构上**，通过解耦视觉编码，解决了理解与生成的粒度冲突，同时保持单一 Transformer 的简洁性。
- **训练上**，三阶段逐步解锁参数，混合理解、文本、生成数据，实现了端到端的统一优化。
- **效果上**，7B 模型在理解和生成两个领域都达到同尺寸开源领先水平。

原生多模态的意义不仅在于「一个模型做两件事」，更在于它展示了**用统一自回归框架处理所有模态的可能性**。无论是 Gemini 的全模态预训练、Qwen3-VL 的长上下文多模态理解，还是 Janus-Pro 的统一理解与生成，都在向同一个方向演进：**让 AI 像人类一样，用同一套感知-认知系统理解世界并创造内容**。未来，随着模型规模、数据规模和 tokenizer 技术的进一步发展，原生多模态有望成为通用人工智能（AGI）感知与创造统一的基础架构之一。
