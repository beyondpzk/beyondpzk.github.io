# 多 Agent 数据工程流水线架构分析

这是一个非常典型的 **"监督者-工作者"（Supervisor-Worker）多 Agent 协作架构**，在数据工程领域特别实用。本文将拆解这个案例的设计逻辑、技术要点和潜在挑战。

## 一、架构模式：分层指挥型（Hierarchical Supervision）

```
┌─────────────────────────────────────┐
│         🎯 指挥 Agent               │
│      (Orchestrator / Supervisor)    │
│   任务分解 │ 状态监控 │ 异常调度      │
└──────────┬──────────┬──────────────┘
           │          │
    ┌──────┴──┐  ┌────┴────┐
    ▼         ▼  ▼         ▼
┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐
│ 采集  │ │ 标注  │ │ 验收  │ │ 存储  │
│ Agent │ │ Agent │ │ Agent │ │ Agent │
└───────┘ └───────┘ └───────┘ └───────┘
```

### 核心设计哲学
- **单一职责**：每个 Agent 只精通一件事（采集只负责爬，标注只负责标）
- **解耦协作**：通过标准化接口传递数据，Agent 之间不直接通信
- **集中管控**：指挥 Agent 掌握全局状态，避免"各自为战"

## 二、五个 Agent 的角色拆解

### 🎯 指挥 Agent（Supervisor）
**核心职责**：不是亲自干活，而是**确保流程正确推进**

| 功能 | 具体行为 |
|------|----------|
| **任务分解** | 将"构建某某数据集"拆分为：采集1000条 → 标注 → 验收 → 存储 |
| **状态机管理** | 维护流水线状态：`WAITING` → `CRAWLING` → `LABELING` → `QC` → `STORING` → `DONE` |
| **异常调度** | 采集Agent失败 → 重试/换源；验收不通过 → 打回标注Agent返工 |
| **资源协调** | 控制并发量，避免存储Agent还没存完，采集Agent又塞过来一批 |

### 1️⃣ 数据采集 Agent
**输入**：目标描述（如"爬取知乎关于AI Agent的高赞回答"）  
**输出**：原始数据（JSON/CSV/图片/文本）

**关键能力**：
- 工具调用：Selenium/Playwright/Scrapy/Requests
- 反爬策略：自动切换IP、调整请求频率、识别验证码
- 自适应：发现目标网站结构变化时，尝试重新解析或上报指挥Agent

### 2️⃣ 数据标注 Agent
**输入**：原始数据  
**输出**：带标签的数据

**实现方式通常有两种**：
- **LLM 自标注**：用大模型直接打标签（如情感分类、实体识别）
- **人机结合**：LLM 预标注 → 人工抽检（但在这个全自动流程里，可能纯靠LLM）

**关键能力**：
- 理解标注规范（指挥Agent下发的标准）
- 一致性校验：同一类数据标注结果不能前后矛盾

### 3️⃣ 数据验收 Agent（QC / Quality Control）
**输入**：已标注数据  
**输出**：通过/不通过 + 原因

**这是整个系统的"质量闸门"**：
- **规则验收**：格式检查、必填项、标签合法性
- **逻辑验收**：标注结果与原文是否矛盾（如文本明明是负面，标成了正面）
- **抽样验收**：随机抽查10%进行交叉验证
- **不通过时**：生成"返工单"，指明问题类型，发回指挥Agent

### 4️⃣ 数据存储 Agent
**输入**：验收通过的数据  
**输出**：存储确认 + 元数据记录

**职责**：
- 写入数据库/对象存储/向量库
- 生成元数据（来源URL、采集时间、标注版本、质量评分）
- 去重处理：避免重复数据入库
- 版本管理：支持数据集的版本回溯

## 三、数据流转与协作流程

```
Step 1: 用户指令
    ↓
"帮我构建一个'电商产品评论情感分析'数据集，需要5000条"

Step 2: 指挥Agent制定计划
    ↓
"采集Agent：从某电商平台爬5000条评论
 标注Agent：标注为正面/负面/中性
 验收Agent：抽检准确率>95%
 存储Agent：存入PostgreSQL"

Step 3: 流水线执行（状态驱动）
┌──────────────────────────────────────────┐
│  采集Agent ━━爬取5000条━━━━━━━━━━━━━━┐  │
│     ↓ 原始数据                        │  │
│  标注Agent ━━打标签━━━━━━━━━━━━━━━━━┐│  │
│     ↓ 标注数据                       ││  │
│  验收Agent ━━抽检：发现200条矛盾━━━━┐││  │
│     ↓ 返工指令（不通过）            │││  │
│  指挥Agent ━━调度：200条回标注Agent重标││  │
│     ↓ 重标后再次验收                ││  │
│  验收Agent ━━通过                    │  │
│     ↓                              │  │
│  存储Agent ━━入库 + 记录元数据━━━━━━┘  │
└──────────────────────────────────────────┘

Step 4: 指挥Agent汇报结果
    ↓
"已完成：入库4800条（去重200条），准确率97%，存储于..."
```

## 四、关键技术实现要点

### 1. 通信协议（Agent 之间如何"对话"）
通常采用 **结构化消息** 而非自由文本：

```json
{
  "from": "采集Agent",
  "to": "指挥Agent",
  "message_type": "TASK_COMPLETE",
  "payload": {
    "batch_id": "batch_001",
    "data_count": 5000,
    "data_location": "/tmp/batch_001.json",
    "quality_hint": "部分页面反爬，实际获取4800条"
  }
}
```

### 2. 指挥Agent的决策逻辑
指挥Agent本质上是一个**状态机 + LLM 决策器**：

```python
# 伪代码示意
def supervisor_loop():
    state = "INIT"
    while state != "DONE":
        if state == "INIT":
            assign_task("采集Agent", config)
            state = "CRAWLING"
        
        elif state == "CRAWLING" and 采集Agent.report_done():
            assign_task("标注Agent", 原始数据)
            state = "LABELING"
        
        elif state == "LABELING" and 标注Agent.report_done():
            assign_task("验收Agent", 标注数据)
            state = "QC"
        
        elif state == "QC":
            result = 验收Agent.get_result()
            if result.pass_rate > 0.95:
                assign_task("存储Agent", result.data)
                state = "STORING"
            else:
                assign_task("标注Agent", result.failed_items, 指令="重点修正")
                state = "LABELING"  # 回退状态
        
        elif state == "STORING" and 存储Agent.report_done():
            state = "DONE"
```

### 3. 记忆与上下文管理
- **短期记忆**：当前批次的处理状态（存在指挥Agent的上下文中）
- **长期记忆**：历史批次的问题模式（如"某网站经常反爬" → 下次自动调整策略）
- **共享存储**：各Agent不直接传数据，而是通过**共享存储区**（如Redis/S3）交换，消息只传指针

## 五、这种架构的优势

| 优势 | 说明 |
|------|------|
| **🔄 可替换性** | 想换采集框架？只改采集Agent，其他不动 |
| **🛡️ 容错性** | 单点故障不影响全局（标注挂了，采集可继续下一批） |
| **📈 可扩展性** | 瓶颈在标注？直接扩容3个标注Agent实例 |
| **🧪 可测试性** | 每个Agent可独立单元测试 |
| **📝 可审计性** | 指挥Agent有完整日志，出了问题可追溯 |

## 六、潜在挑战与优化方案

### ⚠️ 挑战1：指挥Agent成为单点瓶颈
**问题**：所有决策都经过指挥Agent，量大时忙不过来  
**优化**：  
- 指挥Agent只负责"异常决策"，正常流程各Agent自动推进（类似流水线令牌机制）
- 或引入"子指挥"：每1000条数据配一个子指挥，总指挥只管子指挥

### ⚠️ 挑战2：验收标准模糊
**问题**：LLM做验收，标准不一致（今天严明天松）  
**优化**：  
- 验收Agent使用**确定性规则引擎**（如正则、 schema校验）+ LLM辅助判断
- 建立"黄金标准集"：100条人工标注的基准数据，定期自动测试验收Agent的稳定性

### ⚠️ 挑战3：数据在Agent间传递的延迟
**问题**：5000条数据通过消息传递，上下文爆炸  
**优化**：  
- Agent间**不传数据，传指针**（数据存在共享存储，消息只传URI）
- 大文件用对象存储（S3/OSS），Agent只操作元数据

### ⚠️ 挑战4：死循环（标注→验收不通过→返工→验收不通过...）
**问题**：质量不达标，无限返工  
**优化**：  
- 指挥Agent设置**最大重试次数**（如最多返工3次）
- 超过阈值后：降级接受（标记为低置信度）或转人工介入

## 七、这个案例的启发

这个项目的核心智慧在于：**把"数据工程流水线"映射为"组织协作结构"**。

就像真实公司里：
- 有专门爬数据的工程师
- 有标注团队
- 有QA团队
- 有DBA管存储
- 有项目经理协调进度

**Agent 的本质不是替代单个工具，而是模拟一个具备目标、分工、协作、纠错能力的"微型组织"**。
