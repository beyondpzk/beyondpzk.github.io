# SmolVLM：小而强的端侧视觉语言模型

> **论文**：*SmolVLM: Redefining small and efficient multimodal models*  
> **作者**：Andrés Marafioti, Orr Zohar, Miquel Farré, Merve Noyan, Elie Bakouch 等（Hugging Face）  
> **发表时间**：2025-04-07  
> **arXiv**：https://arxiv.org/abs/2504.05299  
> **模型权重与代码**：https://huggingface.co/collections/smol-research/smolvlm-256m-500m-2b-67b78f957e70f0f01b40fd18

---

## 摘要

SmolVLM 是 Hugging Face 推出的一系列**紧凑型视觉-语言模型（Vision-Language Model, VLM）**，目标是在资源受限的端侧和边缘设备上实现高效的多模态推理。该系列包含 **256M、500M 和 2.2B** 三个参数规模，其中最小的 **SmolVLM-256M** 推理时仅需 **<1GB GPU 内存**，却在多项基准上超过了 300 倍大的 **Idefics-80B**。

SmolVLM 的核心设计哲学是：把为大模型设计的架构直接缩小并不是最优解，而应针对小模型的计算和内存约束重新设计。其关键创新包括：
1. **更小的视觉编码器 + 紧凑语言模型** 的参数平衡；
2. **Pixel Shuffle** 视觉 token 压缩与 **图像切块** 策略；
3. **学习的位置 token** 与结构化 prompt；
4. **端到端开源**：权重、数据、训练代码、移动端 Demo 全部公开。

---

## 一、背景：为什么需要「小」VLM？

近年来，GPT-4V、Qwen2-VL、InternVL 等大型 VLM 在图文理解、OCR、文档问答等任务上取得了惊人进展，但它们通常需要数十 GB 的 GPU 内存，难以部署在手机、机器人、AR 眼镜等端侧设备上。

现有的小 VLM（如 PaliGemma、MiniCPM-V、Moondream2）虽然在体积上有所压缩，但很多仍然沿用了大模型的设计选择，例如：
- 使用大尺寸视觉编码器，导致视觉 token 数量爆炸；
- 图像分辨率提升时上下文长度迅速增长；
- 训练数据直接复用 LLM 的 SFT 数据，造成负迁移。

SmolVLM 的出发点是：**小模型需要一套专门的「瘦身」配方**，而不是大模型的等比例缩小。

---

## 二、模型架构

SmolVLM 采用经典的 **self-attention 式 VLM 架构**：视觉 token 与文本 token 拼接后一起输入语言模型。整体流程如下：

```text
图像/视频帧
   ↓
SigLIP 视觉编码器 → 视觉特征
   ↓
Pixel Shuffle（空间压缩）
   ↓
MLP 投影 → 与文本 token 拼接
   ↓
SmolLM2 语言模型 → 文本输出
```

### 2.1 视觉编码器与语言模型的参数分配

论文系统地比较了不同规模的视觉编码器（SigLIP-B/16，93M；SigLIP-SO-400M，428M）与语言模型（SmolLM2 135M / 360M / 1.7B）的组合，发现一个反直觉的结论：

> **对于紧凑 VLM，使用更小的视觉编码器通常更划算。**

- 当语言模型只有 135M 时，搭配 428M 的大编码器反而会因为参数分配失衡而性能下降；
- 当语言模型为 360M 时，大编码器提升约 11.6%，但参数量增加 66%，性价比不高；
- 只有当语言模型达到 1.7B 时，使用大编码器才物有所值。

因此，SmolVLM-256M 和 SmolVLM-500M 都使用 **SigLIP-B/16（93M）**，而 SmolVLM-2.2B 使用 **SigLIP-SO-400M（428M）**。

### 2.2 Pixel Shuffle：激进的视觉 token 压缩

视觉 token 数量直接决定推理时的内存和计算开销。SmolVLM 采用 **Pixel Shuffle（空间到深度的重排）** 来压缩视觉 token：

- 将空间特征图上的像素重新排列到通道维度，从而把空间分辨率降低、通道数增加；
- shuffle 比例 `r=4` 时，token 数量减少为原来的 `1/16`；
- 小模型能从更激进的压缩中受益，因为注意力开销大幅降低；
- 作为代价，极度细粒度的空间定位能力会略有损失，因此 OCR 等任务需要配合图像切块策略。

### 2.3 高分辨率图像与视频处理

- **图像切块（Image Splitting）**：对于高分辨率文档图，SmolVLM 将其切分为多个子图，并保留一个下采样后的全局图，一起送入模型。这样既保留了全局上下文，又获得了局部细节。
- **视频帧处理**：SmolVLM 不支持帧平均（frame averaging），因为实验发现帧平均会显著降低视频理解性能。相反，它直接对视频帧进行缩放后逐帧编码。

### 2.4 长上下文支持

为了让更多视觉 token 能够放入语言模型，SmolVLM 扩展了 RoPE 基频：
- SmolVLM-2.2B 支持 **16k** 上下文；
- 256M / 500M 版本支持 **8k** 上下文。

作为对比，原始 SmolLM2 只有 2k 上下文，无法容纳一张 512×512 图像经 SigLIP 编码后的 1024 个视觉 token。

## 三、训练策略与数据

SmolVLM 的训练分为两个阶段：
1. **Vision Stage**：图文预训练与指令微调；
2. **Video Stage**：加入视频数据进行微调。

### 3.1 视觉阶段数据组成

训练数据强调视觉与结构化数据理解，同时保留少量文本任务以保持语言能力：

| 数据类型 | 内容 |
|---|---|
| 文档理解 | DocVQA、OCR 相关数据 |
| 图像描述 | 多维度 caption 数据 |
| 视觉问答 | 包含约 2% 的多图推理任务 |
| 图表理解 | ChartQA、表格理解等 |
| 视觉推理 | 数学、逻辑类视觉问题 |
| 文本知识问答 | 少量通用知识与推理数据 |

### 3.2 视频阶段数据组成

视频微调阶段保持约 **14% 的文本数据** 和 **33% 的视频数据**，来源包括：

- 视觉描述：LLaVA-video-178k、Video-STAR、Vript、ShareGPT4Video；
- 时序理解：Vista-400k；
- 叙事理解：MovieChat、FineVideo；
- 多图数据：M4-Instruct、Mammoth。

### 3.3 关键训练发现

论文在小模型上总结了一系列实用的训练技巧：

| 设计选择 | 结论 |
|---|---|
| **视觉编码器大小** | 小语言模型配小编码器更划算；大编码器只有在大 LM 上才值得 |
| **上下文长度** | 扩展上下文对紧凑 VLM 收益显著，2k → 16k 带来明显提升 |
| **Pixel Shuffle** | 小模型适合更激进的 `r=4` 压缩，大幅降低 attention 开销 |
| **图像切块** | 提升高分辨率图像和文档任务性能 |
| **帧平均** | 对视频理解有害，SmolVLM 不使用 |
| **位置 token** | 学习的位置 token 比字符串位置（如 `<row_1_col_2>`）更稳定，能避免「OCR loss plague」 |
| **结构化 prompt** | system prompt + 媒体 intro/outro token 能显著提升视频任务表现 |
| **User prompt masking** | SFT 时只训练 completion，mask 掉用户问题可防止过拟合 |
| **复用 LLM-SFT 文本数据** | 直接复用 SmolTalk 等 LLM SFT 数据反而损害性能，因为多样性下降；应保持约 14% 的文本比例 |
| **CoT 数据** | 0.02%–0.05% 的 CoT 数据有益，过量会挤占视觉能力 |
| **视频长度** | 平均视频长度约 3.5 分钟时收益最佳，再长则边际收益递减 |

## 四、模型变体

| 模型 | 视觉编码器 | 语言模型 | 总参数量 | 单图推理内存 |
|---|---|---|---|---|
| **SmolVLM-256M** | SigLIP-B/16 (93M) | SmolLM2-135M | 256M | **0.8 GB** |
| **SmolVLM-500M** | SigLIP-B/16 (93M) | SmolLM2-360M | 500M | **1.2 GB** |
| **SmolVLM-2.2B** | SigLIP-SO-400M (428M) | SmolLM2-1.7B | 2.2B | **4.9 GB** |

## 五、实验结果

### 5.1 图像与多任务基准

| 基准 | 任务 | 256M | 500M | 2.2B | 对比基线 |
|---|---|---:|---:|---:|---|
| OCRBench | 字符识别 | 52.6 | 61.0 | 72.9 | 54.7 (MolmoE-A1B-7B) |
| AI2D | 科学图示 | 46.4 | 59.2 | 70.0 | 71.0 (MolmoE-A1B-7B) |
| ChartQA | 图表理解 | 55.6 | 62.8 | 68.7 | 48.0 (MolmoE-A1B-7B) |
| TextVQA | 图中文字理解 | 50.2 | 60.2 | 73.0 | 61.5 (MolmoE-A1B-7B) |
| DocVQA | 文档问答 | 58.3 | 70.5 | 80.0 | 77.7 (MolmoE-A1B-7B) |
| ScienceQA | 高中科学问答 | 73.8 | 80.0 | 89.6 | 87.5 (MolmoE-A1B-7B) |
| MMMU | 大学多学科 | 29.0 | 33.7 | 42.0 | 33.9 (MolmoE-A1B-7B) |
| MathVista | 数学视觉理解 | 35.9 | 40.1 | 51.5 | 37.6 (MolmoE-A1B-7B) |
| MMStar | 多学科推理 | 34.6 | 38.3 | 46.0 | 43.1 (MolmoE-A1B-7B) |

### 5.2 视频基准

| 基准 | 任务 | 256M | 500M | 2.2B | 对比基线 |
|---|---|---:|---:|---:|---|
| Video-MME | 通用视频理解 | 33.7 | 42.2 | 52.1 | 45.0 (InternVL2-2B) |
| MLVU | 电影 QA + 视频描述 | 40.6 | 47.3 | 55.2 | 48.2 (InternVL2-2B) |
| MVBench | 多视角推理 | 32.7 | 39.7 | 46.3 | 60.2 (InternVL2-2B) |
| WorldSense | 时序 + 物理 | 29.7 | 30.6 | 36.2 | 32.4 (Qwen2-VL-7B) |
| TempCompass | 时序理解 | 43.1 | 49.0 | 53.7 | 53.4 (InternVL2-2B) |

### 5.3 关键结论

1. **效率惊人**：SmolVLM-256M 仅用 0.8GB 内存，平均性能超过 80B 的 Idefics；SmolVLM-2.2B 仅用 4.9GB 内存，即可与 InternVL2-2B（10.5GB）、Qwen2-VL-2B（13.7GB）竞争。
2. **Scaling 有效**：从 256M 到 2.2B，各项任务几乎单调提升，说明架构和数据配方具有良好的扩展性。
3. **视频能力突出**：SmolVLM-2.2B 在 Video-MME 和 WorldSense 上超过了许多更大模型，证明小模型也能做高质量视频理解。
4. **内存效率优于参数效率**：论文指出，对于 VLM，**运行内存（RAM/VRAM）是比参数量更真实的成本指标**。

## 六、端侧部署性能

SmolVLM 不仅在基准上表现好，更强调真实设备上的可部署性：

- **NVIDIA A100**：256M 模型 batch=1 时约 0.8 samples/s，batch=64 时可达 16.3 samples/s；
- **NVIDIA L4**：256M 在 batch=8 时达到峰值 2.7 samples/s；
- **MacBook Pro (M4 Max)**：通过 WebGPU 在浏览器中运行 256M 模型，解码速度可达 **80 tokens/s**；
- **移动端**：HuggingSnap App 可在消费级手机上本地运行 SmolVLM，实现拍照问答、物体检测等交互。

论文还提供了优化的 ONNX 导出，便于跨平台部署。

## 七、下游应用

SmolVLM 发布后已被社区广泛应用于多个场景：

- **ColSmolVLM**：面向端侧设备的多模态推理，支持手机和浏览器部署；
- **Smol Docling**：256M 参数的文档处理专用模型，支持端到端多模态文档转换；
- **BioVQA**：生物医学视觉问答，用于辅助医学影像解读。

## 八、局限与未来方向

### 8.1 局限

1. **语言推理上限**：在 MMMU、AI2D 等强语言推理任务上，SmolVLM 仍落后于大语言骨干更强的模型；
2. **细粒度定位**：激进的 pixel shuffle 会损失部分空间精度，需要依赖图像切块来缓解；
3. **视频帧数限制**：虽然能做视频理解，但超长视频（小时级）仍受上下文长度限制；
4. **数据规模**：相比闭源大模型，训练数据规模和多样性仍有差距。

### 8.2 未来方向

1. **更激进的量化与蒸馏**：进一步降低端侧内存和功耗；
2. **多模态 Agent**：将 SmolVLM 作为视觉感知模块，接入端侧 Agent 系统；
3. **实时视频流**：结合流式编码和缓存机制，支持更长的视频输入；
4. **领域特化模型**：借鉴 Smol Docling 的思路，针对医疗、工业、教育等场景训练专用小模型。

---

## 九、总结

SmolVLM 证明了：**VLM 不一定越大越好，针对端侧场景重新设计的小模型同样可以「小而强」**。通过合理的编码器-LM 参数分配、激进的 token 压缩、图像切块、学习的位置 token 和精心筛选的训练数据，SmolVLM 在 256M 到 2.2B 的参数范围内实现了优异的图文/视频理解能力，同时把推理内存压缩到了 0.8GB–4.9GB。

对于希望在手机、机器人、AR 眼镜等边缘设备上部署多模态能力的开发者来说，SmolVLM 是一个极具吸引力的开源选择。
