FSDP

一句话理解就是: 模型参数太大了, 比如中间一个Weight的参数是 1024维的, 2个gpu, 每个gpu上面存固定的512维, 真正用到的时候再 all_gather, 不用的时候就各自维护自己的那一份; 而且用的时候 all_gather完计算完之后马上释放掉,进入别的参数的forward或者是参数更新.

FSDP（Fully Sharded Data Parallel）详解（PyTorch）

FSDP（Fully Sharded Data Parallel，全切分数据并行）是 PyTorch 分布式训练中用于大模型训练与显存优化的核心技术之一。它通过把模型训练所需的关键状态（参数、梯度、优化器状态等）在多个 GPU 之间进行切分（shard），显著降低单卡显存占用，从而支持更大规模的模型与更大的 batch size。

1. 训练大模型的显存瓶颈来自哪里？

以标准的训练（以 Adam 为例）为例，单卡通常需要存储：

  1. 模型参数（Parameters）
  2. 梯度（Gradients）
  3. 优化器状态（Optimizer States）
    • Adam 通常有 m（一阶动量）和 v（二阶动量），各与参数同尺寸
  4. 激活值（Activations）
    • 用于反向传播，随 batch/序列长度/网络结构而变化
  5. 其他开销（通信 buffer、临时张量等）

在传统 DDP（DistributedDataParallel） 中，每张 GPU 都保留一份完整参数 + 完整优化器状态，只是每张卡处理不同数据并在反向时做梯度 AllReduce。因此 DDP 的显存随模型规模增长非常快。

2. FSDP 的核心思想

FSDP 的核心是把“模型训练状态”做完全切分：

  • 参数（Parameters）切分
  • 梯度（Gradients）切分
  • 优化器状态（Optimizer states）切分

也就是：每张 GPU 只持有这些状态的一部分。在需要计算某一层时，再通过通信把该层需要的完整参数临时聚合到本卡进行计算；算完后再释放/重新切分。

一句话总结：

平时各卡只存 shard；用到某层时 all-gather 得到完整参数；算完再回收。

3. FSDP 与 DDP / ZeRO 的关系

3.1 DDP

  • 参数：复制（每卡一份）
  • 梯度：反向后 AllReduce（每卡仍持有完整梯度）
  • 优化器状态：复制（每卡一份）
  • 优点：简单稳定
  • 缺点：显存占用大

3.2 ZeRO（DeepSpeed）分阶段切分

ZeRO 提出把训练状态逐步切分：

  • ZeRO-1：切分优化器状态
  • ZeRO-2：再切分梯度
  • ZeRO-3：再切分参数（最省显存）

FSDP 在理念上接近 ZeRO-3：对参数/梯度/优化器状态进行全切分，并用通信在前向/反向阶段按需聚合参数。

区别（粗略理解）：

  • FSDP 是 PyTorch 原生体系（torch.distributed.fsdp），与 PyTorch 编译器、检查点、混合精度等更自然集成。
  • DeepSpeed ZeRO 是一套更“完整系统”，功能更丰富（如更成熟的 offload、pipeline 等），但集成方式不同。

4. FSDP 训练时到底发生了什么？（按阶段拆解）

设有 N 张 GPU。参数张量 W 会被切分成 N 份：W = [W0, W1, ..., W(N-1)]，每张卡只常驻其中一份。

4.1 前向（Forward）

对某一层来说：

  1. AllGather 参数：
    该层的 shard 参数在各卡间进行 all_gather，拼成完整 W_full
  2. 用 W_full 完成前向计算，得到激活输出
  3. （可选）释放该层完整参数，只保留 shard，降低峰值显存
    • 这一步称为类似“参数重分片（reshard）”策略

4.2 反向（Backward）

反向时通常还要用到该层参数：

  1. 再次 AllGather 参数（如果前向后释放了完整参数）
  2. 计算梯度 dW_full
  3. ReduceScatter 梯度：
    将完整梯度按 shard 切回去，并做跨卡归并，最终每卡只留下自己的 dW_shard

4.3 优化器更新（Optimizer Step）

优化器状态也是 shard 的：

  • 每张卡用自己的 W_shard、dW_shard、state_shard 做更新
  • 更新后的参数仍然是 shard 形式驻留

5. 为什么 FSDP 能显著省显存？

在 DDP 下，每卡持有完整参数/梯度/优化器状态；在 FSDP 下，它们都约按 1/N 缩小（理想情况下）。

以 Adam 为例（非常粗略的数量级）：

  • DDP 每卡参数：P
  • 梯度：P
  • Adam 状态：2P
  • 合计（不含激活）：约 4P

FSDP（全切分）后，每卡约：

  • 参数：P/N
  • 梯度：P/N
  • Adam 状态：2P/N
  • 合计：约 4P/N

这就是为什么 GPU 数越多，FSDP 对大模型越友好（当然通信开销也会上升）。

6. FSDP 的关键概念与配置点

6.1 Auto Wrap（自动包裹策略）

FSDP 通常不是把整个模型当作一个大块来 shard，而是把模型按模块（module）进行包裹（wrap）。包裹粒度影响：

  • 通信频率（wrap 太细 → all-gather 次数多）
  • 显存峰值（wrap 合理 → 峰值更低）
  • 性能（需要平衡通信与计算）

典型策略：

  • 以 Transformer block 为单位 wrap（常见且有效）
  • 或者按参数量阈值 wrap（大于一定参数量的层才 wrap）

6.2 Mixed Precision（混合精度）

FSDP 常配合 bf16/fp16：

  • 参数通信可以用低精度
  • 计算用 bf16
  • 优化器可能保留 fp32 master weights（看策略）

混合精度能进一步降低显存与提升吞吐，但要注意数值稳定性。

6.3 CPU Offload（可选）

可把部分状态（如优化器状态）offload 到 CPU：

  • 更省 GPU 显存
  • 但会引入 PCIe/NVLink 传输开销，通常更慢

6.4 Activation Checkpointing（激活重计算）

FSDP 解决的是“参数/梯度/优化器状态”的显存压力；激活往往仍是大头。常与 activation checkpointing 组合：

  • 少存激活，多做重算
  • 进一步降低峰值显存，代价是算力增加（训练变慢）

7. FSDP 的通信模式与性能影响

FSDP 的主要通信：

  • AllGather（参数）：前向/反向前需要完整参数
  • ReduceScatter（梯度）：反向结束时将梯度切分并规约

性能关键点：

  1. wrap 粒度：粒度过细会导致 all-gather 次数过多
  2. 重叠（overlap）：通信与计算能否 overlap（取决于实现与网络）
  3. 网络拓扑与带宽：NVLink / IB 对性能影响极大
  4. 层次化通信：多机多卡可能需要更复杂的通信调度

经验上：

  • 单机多卡（NVLink）更容易得到好的性能
  • 多机训练对通信更敏感，需要更 careful 的 wrap 和 batch 配置

8. FSDP 的检查点（Checkpoint）与保存/加载

FSDP 下参数是 shard 的，所以保存模型时有两种常见方式：

  1. Sharded checkpoint（分片保存）
    • 每个 rank 保存自己那份 shard
    • 保存/加载快、占用分布式存储更均衡
    • 推理或单机加载时需要合并或使用相应加载逻辑
  2. Full state dict（合并保存）
    • 聚合所有 shard 得到完整权重再保存
    • 更通用，但保存时需要额外通信与内存

通常大模型推荐 sharded checkpoint。

9. 适用场景与不适用场景

9.1 适用

  • 模型非常大，DDP 放不下
  • 优化器状态占用高（例如 Adam）
  • 多 GPU 环境下希望显存线性下降
  • 与 activation checkpointing 联合，最大化模型规模

9.2 不太适用/收益不大

  • 模型很小（通信开销可能超过收益）
  • GPU 数很少但网络/带宽弱（通信成为瓶颈）
  • 强依赖特定自定义 CUDA op、且与 FSDP wrap/重分片不兼容（需要验证）

10. 实战建议（经验性总结）

  1. 优先以 Transformer block 为 wrap 单元
    • 通信次数少、效果稳定
  2. bf16 通常比 fp16 更稳（支持的话）
  3. 搭配 activation checkpointing
    • 大模型训练几乎标配
  4. 关注峰值显存而不是平均显存
    • all-gather 会产生瞬时峰值
  5. 多机训练要特别关注通信
    • 合理的 batch/sequence、梯度累积、wrap 粒度都很关键
  6. 检查点优先用 sharded state dict
    • 更适合长期训练与断点续训

具体以MLP为例来说时

下面用一个两层 MLP（Linear-ReLU-Linear）作为例子，假设用 2 张 GPU（rank0 / rank1），并且使用 PyTorch FSDP 的典型默认思路：参数全切分 + 反向时 reduce-scatter 梯度 + shard 版优化器更新，详细把“一次迭代里梯度如何产生、如何通信、如何更新参数”讲清楚。

说明：FSDP 有一些策略开关（是否前向后立刻 reshard、是否保留 full param、是否同步/异步 overlap 等）。下面讲的是最常见/最能体现机制的一条主线：平时只存 shard，用到某层就 all-gather，算完就 reshard；梯度用 reduce-scatter 留 shard；优化器在 shard 上更新。

0. 模型与并行方式设定

MLP：

  • Layer1：y = x @ W1^T + b1，W1: [H, D]，b1: [H]
  • ReLU
  • Layer2：o = y @ W2^T + b2，W2: [C, H]，b2: [C]

数据并行（DP）方式：

  • rank0 拿到一个 mini-batch 的一半：x0, t0
  • rank1 拿到另一半：x1, t1

FSDP 切分方式（2 GPU）：

  • 每个参数张量按某个维度被“扁平化后切块”（真实实现里常是 flat parameter + shard），概念上你可以理解为：
    • W1 被切成两半：rank0 保存 W1_shard0，rank1 保存 W1_shard1
    • b1 同理（或因为太小可能不切，但机制一致）
    • W2, b2 同理

本地常驻状态：

  • rank0 常驻：{W1_shard0, b1_shard0, W2_shard0, b2_shard0} 以及对应的 optimizer state shard
  • rank1 常驻：{W1_shard1, b1_shard1, W2_shard1, b2_shard1} 以及对应的 optimizer state shard

1. 一次迭代总览（你要关注的“梯度更新链路”）

一次迭代可以拆成：

  1. 前向：逐层 AllGather 得到 full 参数 → 计算 →（可选）立刻 Reshard 释放 full 参数
  2. 反向：逐层再次 AllGather 得到 full 参数 → 计算得到 full 梯度 → ReduceScatter 让每卡只留下自己那份梯度 shard
  3. 优化器更新：每卡用 (param_shard, grad_shard, opt_state_shard) 做一步更新
  4. 清梯度：清理 shard 梯度

下面按层，把通信与梯度如何“从 full 变 shard”讲细。

2. 前向阶段（Forward）：参数如何被临时“拼起来”用

2.1 进入 Layer1 前：AllGather 参数

  • rank0、rank1 各自持有 W1_shard*
  • 需要做矩阵乘时，每个 rank 都需要完整的 W1_full

于是发生一次通信（概念上）：

  • W1_full = AllGather(W1_shard0, W1_shard1)
  • 同理得到 b1_full

结果：rank0 与 rank1 此刻都各自拥有一份 W1_full, b1_full（临时的）。

2.2 计算本地前向

  • rank0：h0 = relu(x0 @ W1_full^T + b1_full)
  • rank1：h1 = relu(x1 @ W1_full^T + b1_full)

2.3 （常见策略）Reshard：丢掉 full 参数，只保留 shard

为了降低峰值显存，FSDP 通常会在该层 forward 用完后：

  • 释放/丢弃 W1_full, b1_full
  • 只保留常驻的 W1_shard* , b1_shard*

同理 Layer2 forward 也会做：

  • AllGather(W2_shard*) -> W2_full
  • 计算 o0/o1
  • 释放 W2_full

到此为止，你可以把 forward 记成一句话：
每层：all-gather 参数 → 用本地数据做计算 → 立刻释放 full 参数（只留 shard）。

3. 反向阶段（Backward）：梯度如何产生、如何“切回去”并规约

反向从最后一层开始（Layer2 → Layer1）。关键点：

  • 每张卡只用自己的数据计算局部梯度，但最终要形成“等价于全局 batch”的梯度。
  • DDP 用 AllReduce 让每卡拿到同一份 full 梯度；
  • FSDP 用 ReduceScatter：直接把“全局规约后的梯度”切成 shard 分给各卡，避免每卡持有完整梯度。

下面以 Layer2 为例讲清楚。

3.1 Layer2 反向：先临时拿到 full 参数（AllGather）

因为 forward 后通常释放了 W2_full，反向计算 dW2 又需要 W2，所以：

  • W2_full = AllGather(W2_shard0, W2_shard1)
  • b2_full = AllGather(...)

3.2 计算“本地（local）”梯度：每卡只基于自己数据

设损失为 L（例如 CrossEntropy）。反向中每卡会算出基于自己 mini-batch 部分的梯度：

  • rank0 得到：dW2_local0、db2_local0（形状与 full 参数一致）
  • rank1 得到：dW2_local1、db2_local1

直观理解：

  • dW2_local0 只反映 x0,t0 这部分数据的贡献
  • dW2_local1 只反映 x1,t1 这部分数据的贡献

如果你希望等价于“全局 batch = 两卡 batch 拼起来”的梯度，那么需要做跨卡规约：

  • dW2_global = dW2_local0 + dW2_local1（或平均，取决于 loss reduction 设置）

3.3 ReduceScatter：一步完成“规约 + 切分”，只留下梯度 shard

FSDP 不会让每个 rank 都拿到 dW2_global 的完整张量，而是做：

  • dW2_shard0, dW2_shard1 = ReduceScatter(dW2_local0, dW2_local1)

它同时做两件事：

  1. Reduce（规约）：把不同 rank 的 local 梯度相加/求平均，得到全局等价梯度
  2. Scatter（切分分发）：把规约后的 full 梯度按 shard 切开，rank0 得到 shard0，rank1 得到 shard1

最终结果：

  • rank0 最终只保留 dW2_shard0、db2_shard0
  • rank1 最终只保留 dW2_shard1、db2_shard1
  • 并且它们已经是全局等价梯度的对应分片（不是“本地梯度分片”）

这点非常关键：
每个 rank 手里的 grad shard，是“全局规约后梯度”的 shard。
所以接下来每卡用自己的 grad shard 更新自己的 param shard，是正确的。

随后 W2_full 可以释放，回到只保留 shard 的状态。

3.4 Layer1 反向：同样的流程再来一次

Layer1 反向同理：

  1. W1_full = AllGather(W1_shard*)
  2. 本地算 dW1_local*
  3. ReduceScatter(dW1_local*) -> dW1_shard*
  4. 释放 W1_full

4. 优化器更新（Optimizer Step）：每卡只更新自己那一片参数

现在每个 rank 拥有：

  • W1_shard, b1_shard, W2_shard, b2_shard
  • dW1_shard, db1_shard, dW2_shard, db2_shard
  • optimizer state shard（例如 Adam 的 m_shard, v_shard）

以 Adam 为例，在每个 rank 本地独立执行：

  • m_shard = beta1*m_shard + (1-beta1)*grad_shard
  • v_shard = beta2*v_shard + (1-beta2)*(grad_shard^2)
  • param_shard -= lr * m_hat_shard / (sqrt(v_hat_shard) + eps)

这里不需要额外通信，因为：

  • 参数是 shard 的
  • 梯度也是 shard 的（且已是全局等价）
  • 优化器状态也是 shard 的
    => 更新完全可以本地进行。

更新完成后：

  • rank0 得到新的 W*_shard0
  • rank1 得到新的 W*_shard1

下一次迭代 forward 需要该层参数时，再次 AllGather 拼成新的 W*_full 来计算即可。

5. 用“时间线”把一次迭代写成清单（Layer2/Layer1）

假设采用“forward 后 reshard”，一次 step 的核心事件像这样：

Forward

  • Layer1：
    • AllGather(W1_shard) → W1_full
    • forward compute
    • free W1_full（reshard）
  • Layer2：
    • AllGather(W2_shard) → W2_full
    • forward compute
    • free W2_full（reshard）

Backward

  • Layer2：
    • AllGather(W2_shard) → W2_full
    • backward compute → dW2_local
    • ReduceScatter(dW2_local) → dW2_shard（全局规约后的分片）
    • free W2_full
  • Layer1：
    • AllGather(W1_shard) → W1_full
    • backward compute → dW1_local
    • ReduceScatter(dW1_local) → dW1_shard
    • free W1_full

Optimizer step（本地）

  • Adam/SGD 在 shard 上更新参数与状态

6. 和 DDP 梯度更新过程对比一下（帮助你确认理解）

同样 2 GPU：

DDP

  • 参数：每卡完整一份
  • 反向后：每卡得到完整 dW_local
  • AllReduce：每卡得到完整 dW_global
  • 每卡用完整梯度更新完整参数（所以每卡参数始终一致）

FSDP

  • 参数：每卡只有一片 shard
  • 反向时：先本地算 dW_local_fullshape（概念上）
  • ReduceScatter：每卡只保留 dW_global_shard
  • 每卡只更新自己那片参数 shard（参数整体由多卡共同“拼出”）

7. 你可能会追问的两个关键细节（我先点出来）

7.1 “本地算梯度时不是需要 full 参数吗？那不是又占显存了？”

是的，所以 FSDP 才有：

  • 按层 all-gather（而不是一次性把整个模型 all-gather）
  • 用完就 reshard/free full 参数
  • wrap 粒度越合理，峰值越低

7.2 “本地算出来的 dW_local 是 full shape，岂不是反向时也要暂存完整梯度？”

实现上通常会尽量让 full-gradient 的生命周期很短，并且会用通信/流式方式尽早 reduce-scatter，减少峰值；另外 FSDP 常配合混合精度、梯度累积策略等进一步降低压力。

如果你愿意，我也可以把“为什么 backward 仍然能避免完整梯度常驻”按 PyTorch FSDP 的更贴近实现方式（flat param、chunk、prefetch、overlap）展开。

8. 如果你给我两个信息，我可以把过程讲到“完全贴合你要跑的配置”

  1. 你想用的优化器：SGD / AdamW？
  2. 你希望的 FSDP 策略：
    • forward 后是否 reshard（默认通常是开）
    • 是否混合精度 bf16/fp16
    • 是否 activation checkpointing
