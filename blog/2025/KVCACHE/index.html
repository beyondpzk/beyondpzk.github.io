KV-Cache

我理解: 本质上就是随着token数量的增加, 比如prompts有10个token, 接下来开始生成, 生成完一个后变成11个token,在生成12个token的时候, 最初的做法会重新来计算K, V; 但是其实因为那些操作是线性的, 所以本质上其实只需要算新增token的q,v即可. 但是attn的计算并没有省.

KV-Cache（Key/Value Cache）技术原理详解（结合具体例子）

KV-Cache 是 Transformer 自回归生成（逐 token 解码）时的关键加速技术：把历史 token 在注意力层里计算出的 Key/Value 保存起来，后续生成新 token 时不再重复计算历史部分，从而显著降低计算量与延迟。

下面我会用一个非常具体的“逐 token 生成”例子，从注意力计算公式、无缓存的重复计算、KV-Cache 如何复用、缓存张量形状、复杂度与工程细节、常见变体（Paged KV、MQA/GQA、滑窗等）完整讲清楚。

1. 场景：为什么需要 KV-Cache？

在大语言模型推理里，常见模式是：

  • 你给一个 prompt（上下文），模型先“读”完 prompt（prefill）
  • 然后开始一个 token 一个 token 地生成（decode）

问题在 decode 阶段：每生成一个新 token，都要在每一层 self-attention 中，让新 token 去“看”所有历史 token。
如果不做缓存，模型会在每一步把历史 token 的 K/V 反复算一遍，浪费巨大。

KV-Cache 就是为了解决这个重复计算。

2. 先把 Self-Attention 写清楚：Q/K/V 怎么来的？

对某一层 self-attention，输入是每个 token 的隐藏状态 h_t（维度 d_model）：

  • q_t = h_t W_Q
  • k_t = h_t W_K
  • v_t = h_t W_V

对当前 token t 的注意力输出（单头简化）：

[ \text{Attn}(t) = \text{softmax}\left(\frac{q_t K_{1:t}^T}{\sqrt{d_k}}\right) V_{1:t} ]

其中：

  • K_{1:t} 是把从 1 到 t 的所有 key 堆起来的矩阵
  • V_{1:t} 同理
  • 自回归 mask 保证只看过去（1..t）

关键点：在 decode 时，每一步只新增一个 token，因此只新增一行 k_t 和 v_t。历史的 k_1..k_{t-1}, v_1..v_{t-1} 是不变的。

3. 具体例子：用一个很小的 Transformer 来演示 KV-Cache

假设我们有一个极简配置（只为说明机制）：

  • batch = 1
  • num_layers = 2
  • num_heads = 2
  • head_dim = 4
  • 所以 d_model = num_heads * head_dim = 8
  • 输入 prompt 有 3 个 token：[A, B, C]
  • 接下来要生成 2 个 token：[D, E]

我们关注 第 1 层 的注意力（第 2 层完全同理）。

4. 没有 KV-Cache 会发生什么？（重复计算）

4.1 Prefill（处理 prompt：A,B,C）

prefill 通常一次性把 [A,B,C] 喂进去（seq_len=3）：

  • 计算 Q,K,V（都是长度 3）
  • 计算 attention 输出

这一步没问题，因为本来就要算 3 个 token 的全部注意力。

4.2 Decode 第一步：生成 D

现在上下文变成 [A,B,C,D]（长度 4）。

如果没有 KV-Cache，模型往往会再次把整个序列 [A,B,C,D] 作为输入（即便实现上可以做增量，很多朴素实现会重复计算历史），导致：

  • 又计算一遍 k_A,k_B,k_C 和 v_A,v_B,v_C
  • 还要计算 k_D,v_D

这意味着历史 3 个 token 的 K/V 被重复算了。

4.3 Decode 第二步：生成 E

上下文 [A,B,C,D,E]（长度 5），又重复算：

  • k_A..k_D、v_A..v_D 全部重算
  • 新的 k_E,v_E

重复计算越来越严重：生成越长，浪费越大。

5. KV-Cache 的核心做法：缓存历史 token 的 K/V

KV-Cache 的策略是：

  • 在 prefill 处理完 [A,B,C] 后，把每一层、每个头的：
    • K_cache = [k_A, k_B, k_C]
    • V_cache = [v_A, v_B, v_C] 存起来
  • 之后 decode 每一步只算新 token 的 k_new,v_new，并追加到 cache
  • attention 时直接用 K_cache / V_cache 拼上新 token 的 K/V

6. 用同一个例子详细走一遍：有 KV-Cache 时每一步算什么

6.1 Prefill 阶段（输入 A,B,C）

对第 1 层：

  • 输入 hidden states: H = [h_A, h_B, h_C]，shape [1, 3, 8]
  • 得到：
    • K = [k_A, k_B, k_C]，shape [1, 2, 3, 4]（[batch, heads, seq, head_dim]）
    • V = [v_A, v_B, v_C]，shape 同上

把它保存起来：

  • K_cache(layer1) = K
  • V_cache(layer1) = V

对第 2 层也做同样缓存。

注意：KV-Cache 是每一层都有一份，因为每层的 W_K/W_V 不同，算出的 K/V 也不同。

6.2 Decode 步骤 1：生成 D（只输入 D 这一个 token）

此时模型不会把 [A,B,C,D] 全喂进去，而是：

  1. 只拿到新 token D 的隐藏状态 h_D（shape [1,1,8]）
  2. 计算当前层的：
    • q_D, k_D, v_D（只算 1 个 token 的 K/V）
  3. 把 k_D, v_D 追加到 cache：
  • K_cache = concat([k_A,k_B,k_C], [k_D]) -> seq_len=4
  • V_cache 同理
  1. 计算 attention（对 token D）：

[ \text{Attn}(D)=\text{softmax}\left(\frac{q_D K_{cache}^T}{\sqrt{d_k}}\right)V_{cache} ]

其中 K_cache/V_cache 已包含 A,B,C,D 的历史与当前。

关键收益：A/B/C 的 k,v 没有重复计算，直接复用缓存。

6.3 Decode 步骤 2：生成 E（只输入 E）

重复上述过程：

  • 只算 q_E,k_E,v_E
  • cache 变为长度 5
  • 用 q_E 去和 K_cache(长度5) 做注意力

7. 复杂度对比：KV-Cache 为什么能大幅提速？

7.1 不用 KV-Cache（朴素重复计算）

生成长度从 1..T，每一步都对长度 t 的序列算 K/V：

  • K/V 计算量：(\sum_{t=1}^{T} O(t)) = O(T^2)
  • attention（QK^T）本身也要做（这是不可避免的部分），但 K/V 重算是额外浪费

7.2 用 KV-Cache（增量计算 K/V）

  • 每一步只算新 token 的 K/V：(\sum_{t=1}^{T} O(1)) = O(T)
  • attention 仍需：每一步 q_t 与 K_cache(1..t) 做点积，所以仍是 (\sum O(t)=O(T^2))

所以 KV-Cache 主要把“重复的 K/V 投影计算”从 O(T^2) 降为 O(T)，并且把大量矩阵乘变成更小的计算，从而显著降低 decode 延迟。

常见误解：KV-Cache 不能把 attention 的整体复杂度从 O(T^2) 变成 O(T)；它主要节省的是K/V 的重复计算（以及相关的中间张量与带宽）。

8. KV-Cache 的张量结构（工程实现最常见的形状）

常见缓存结构（每层一份）：

  • K_cache: [batch, n_heads, seq_len_cached, head_dim]
  • V_cache: [batch, n_heads, seq_len_cached, head_dim]

对自回归推理：

  • prefill 后 seq_len_cached = prompt_len
  • 每生成 1 个 token，seq_len_cached += 1

如果 batch > 1，且各样本长度不同，还需要管理：

  • 不同序列的有效长度
  • padding 或者用更高级的缓存管理（Paged KV）

9. Prefill 与 Decode 的区别：KV-Cache 在两阶段怎么用？

9.1 Prefill（prompt 阶段）

  • 输入 seq_len 可能很长
  • 通常一次性算完所有 token 的 Q/K/V（全矩阵）
  • 输出 logits 的最后一个位置用于开始生成
  • 同时把整个 prompt 的 K/V 填进 cache

特点：

  • 计算更像训练时的“全序列并行”
  • latency 取决于 prompt 长度

9.2 Decode（生成阶段）

  • 每一步输入 seq_len=1（只输入最新 token）
  • 只算该 token 的 Q/K/V
  • 与 cache 做 attention
  • 更新 cache

特点：

  • 每一步都要读取越来越长的 cache（memory bandwidth 重要）
  • KV-Cache 的内存布局、分页、量化影响巨大

10. 代价：KV-Cache 省算力但吃显存（显存如何估算）

KV-Cache 本质是把每层每个 token 的 K/V 存起来，所以显存线性随：

  • batch
  • 生成长度（prompt + generated）
  • 层数
  • heads
  • head_dim
  • 数据类型（fp16/bf16/int8 等）

粗略估算（以 fp16，每元素 2 bytes）：

[ \text{KV bytes} \approx 2(\text{K+V}) \times L \times B \times T \times n_{heads} \times d_{head} \times 2\text{bytes} ]

其中最前面的 2(K+V) 表示 K 和 V 两份。

例如（仅示意）：

  • L=32 层
  • B=1
  • T=4096（上下文长度）
  • n_heads=32
  • d_head=128
  • dtype=fp16（2 bytes）

KV 体积会非常可观，这就是为什么长上下文推理时显存常被 KV-Cache 吃满。

11. 常见关键优化：MQA/GQA、Paged KV、KV 量化、滑窗

11.1 MQA / GQA：减少 KV 头数

在标准多头注意力（MHA）中，K/V 对每个 head 都不同，因此 cache 大小与 n_heads 成正比。

  • MQA（Multi-Query Attention）：所有 query heads 共享同一套 K/V（KV 头数=1）
  • GQA（Grouped-Query Attention）：query heads 分组共享 K/V（KV 头数 < Q 头数）

效果：

  • KV-Cache 显存按 KV 头数比例下降
  • 读取带宽降低，decode 速度通常更好
  • 代价：模型结构变化（训练时就要用）

11.2 Paged KV Cache：把 cache 当“显存分页系统”管理

当你有：

  • 大 batch
  • 很多并发请求
  • 每个请求长度不同、动态增长

如果用简单的连续大张量 append，会导致：

  • 频繁的内存搬运/重分配
  • 碎片
  • 难以高效复用

Paged KV（vLLM 等采用）把 KV 存在固定大小的“块”（page/block）里：

  • 每个序列有一个“页表”指向它占用的块
  • 新 token 到来就追加一个块/或写入块尾
  • 避免频繁拷贝，提升吞吐与并发

11.3 KV 量化（int8/int4）

把 KV-Cache 从 fp16/bf16 压到 int8/int4：

  • 显存大幅下降
  • 带宽压力减轻（常是 decode bottleneck）
  • 但会引入精度损失，需要校准或更复杂的量化策略

11.4 Sliding Window / Chunked Attention：限制能看到的历史长度

如果模型或系统允许只看最近 W 个 token：

  • KV-Cache 只保留最近窗口（或用环形 buffer）
  • 生成很长时显存不再线性增长
  • 代价：模型有效上下文变短（或需要特定位置编码机制支持）

12. 一个“更具象”的数值演示（用 shape 帮你直观理解）

继续用前面的极简例子：

  • heads=2, head_dim=4
  • prompt=3 tokens

Prefill 后（第1层）：

  • K_cache: [1, 2, 3, 4]
  • V_cache: [1, 2, 3, 4]

Decode 生成 D 后：

  • 新的 k_D: [1, 2, 1, 4]
  • concat 后：
    • K_cache: [1, 2, 4, 4]
    • V_cache: [1, 2, 4, 4]

计算注意力时（对 D）：

  • q_D: [1, 2, 1, 4]
  • K_cache^T: 在实现里会变成适合 matmul 的布局
  • q_D @ K_cache^T 的结果形状大致为：
    • attention scores: [1, 2, 1, 4]（每个 head 对 4 个历史位置一个 score）

再 softmax，再乘 V_cache 得到：

  • output: [1, 2, 1, 4]（再 reshape 回 [1,1,8]）

生成 E 时同理，只是 cache 的 seq_len 变成 5。

13. 总结：一句话概括 KV-Cache 的原理（但这里仍给你完整要点）

  • 缓存内容：每层、每个 token 的 Key/Value（不是 Q）
  • 缓存时机：prefill 算完 prompt 后存；decode 每步把新 token 的 K/V 追加
  • 复用方式：下一步生成只算新 token 的 Q/K/V，用新 Q 去和缓存 K 做注意力，再用权重加权缓存 V
  • 收益来源：避免重复计算历史 K/V，降低 decode 计算与延迟
  • 代价：显存/带宽占用随长度线性增长，需要 MQA/GQA、Paged KV、量化、滑窗等优化
