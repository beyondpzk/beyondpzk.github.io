# verl

Verl 的设计目标非常明确：**解决大模型（LLM）强化学习（RLHF）在“灵活性”与“高性能”之间的矛盾。**

为了实现这一点，Verl 引入了一套被称为 **HybridFlow (混合流)** 的核心架构。

---

### 一、 顶层设计哲学：HybridFlow (混合流)

在传统的深度学习框架中，我们要么选择 **灵活性**（如 HuggingFace TRL，代码简单但难以扩展到多机多卡），要么选择 **高性能**（如 Megatron-LM，性能极强但代码极度复杂，难以修改算法）。

Verl 的 **HybridFlow** 架构将系统拆分为两个完全解耦的层面：

1.  **资源编排层 (Resource Orchestration Layer):**
    *   **工具:** **Ray**。
    *   **职责:** 负责管理 GPU 资源、启动进程、在不同节点间传输数据。
    *   **特点:** 极度灵活。你可以像写 Python 脚本一样调度成百上千个 GPU。
2.  **计算执行层 (Computation Execution Layer):**
    *   **工具:** **PyTorch FSDP** 或 **Megatron-LM**。
    *   **职责:** 负责单个模型内部的高性能计算（前向/反向传播、梯度同步）。
    *   **特点:** 极度高效。利用底层通信原语（NCCL）榨干硬件性能。

**一句话总结：** Verl 用 Ray 来指挥，用 FSDP/Megatron 来干活。

---

### 二、 核心架构组件 (The Core Components)

Verl 的架构可以看作是一个 **“单控制器 - 多工作组” (Single-Controller, Multi-WorkerGroup)** 的系统。

#### 1. Trainer (控制器 / 大脑)
*   **代码位置:** `verl/trainers/ray_trainer.py` (基类) / `verl/trainers/ppo_trainer.py`
*   **角色:** 它是整个训练任务的总指挥。
*   **运行位置:** 通常运行在 **CPU** (Driver Node) 上。
*   **职责:**
    *   它**不持有**任何模型权重。
    *   它**不进行**任何矩阵运算。
    *   它维护 RL 算法的状态机（State Machine）。例如：它决定什么时候让 Actor 生成数据，什么时候让 Critic 更新参数。
    *   它通过 Ray 的 API 向远端的 Worker 发送指令。

#### 2. WorkerGroup (工作组 / 躯干)
这是 Verl 最核心的抽象概念。
*   **定义:** 一个 `WorkerGroup` 代表了 **“一组 GPU 资源 + 一个模型实体”**。
*   **抽象:** 对 Trainer 来说，`WorkerGroup` 就像是一个本地对象，Trainer 可以直接调用 `worker_group.generate()` 或 `worker_group.update()`。但实际上，这个调用会被分发到背后的几十张 GPU 上并行执行。
*   **RL 中的四大金刚:** 在 PPO 算法中，Verl 通常会实例化四个 `WorkerGroup`：
    1.  **Actor:** 策略模型（生成文本）。
    2.  **Critic:** 价值模型（评估状态价值）。
    3.  **Ref (Reference):** 参考模型（用于计算 KL 散度，防止模型跑偏）。
    4.  **Reward:** 奖励模型（给生成的文本打分）。

#### 3. Worker (工人 / 细胞)
*   **代码位置:** `verl/workers/fsdp_workers.py` 或 `megatron_workers.py`
*   **运行位置:** 每一张 **GPU** 上都有一个 Worker 实例。
*   **职责:**
    *   **持有权重:** 真正的 LLM 模型权重在这里加载。
    *   **执行计算:** 执行 `forward`, `backward`, `optimizer.step()`。
    *   **并行通信:** Worker 之间通过 NCCL 进行梯度同步（FSDP/Megatron 逻辑）。

#### 4. ResourcePool (资源池 / 硬件抽象)
*   **职责:** 管理物理硬件。
*   **灵活性:** Verl 允许你定义不同的资源池。
    *   *例子:* 你可以让 Actor 独占 8 个节点（64卡），而让 Reward Model 只用 2 个节点（16卡）。
    *   *资源复用 (Collocation):* Verl 支持将 Actor 和 Ref 模型放在同一个资源池中（同一组 GPU）。通过分时复用（先算 Actor，再算 Ref）来节省显存。

---

### 三、 关键数据流转：DataProto

在分布式系统中，数据传输往往是瓶颈。Verl 设计了 `DataProto` 来解决这个问题。

*   **Smart Batching:** 它不仅仅是一个 Tensor 字典。它支持 Tensor（如 `input_ids`）和非 Tensor 数据（如原始 Prompt 文本、Meta 信息）的混合打包。
*   **Zero-Copy (零拷贝):**
    *   当 Actor 生成数据后，数据**不会**全部传回 Trainer (CPU)。
    *   数据被存储在 Ray 的 **Object Store (共享内存)** 中。
    *   Trainer 只拿到一个 **ObjectRef (数据句柄)**。
    *   当 Trainer 把这个句柄传给 Reward Model 时，Reward Model 的 Worker 直接从共享内存读取数据。这避免了 CPU 和 GPU 之间昂贵的数据搬运。

### 四、 深度解析：一次 PPO 训练步的生命周期

为了彻底理解架构，我们来看看最新版 Verl 是如何执行一步 PPO 训练的。

**场景:** 我们有 Actor (A), Critic (C), Reward (R), Ref (Ref) 四个模型。

#### 阶段 1: 采样 (Rollout)
1.  **Trainer** 发出指令: `actor_wg.generate_sequences(prompts)`。
2.  **Actor WorkerGroup** 接收指令。
    *   *黑科技:* 如果配置了 **vLLM**，Worker 会将 PyTorch 权重共享给 vLLM 引擎，利用 vLLM 的 PagedAttention 进行极速推理。
3.  **Actor** 生成 `(Prompt, Response)`，打包成 `DataProto`，存入 Ray Object Store。
4.  **Trainer** 收到数据句柄。

#### 阶段 2: 评估 (Evaluation)
1.  **Trainer** 将数据句柄分别发给 **Reward WG**, **Critic WG**, **Ref WG**。
2.  **并行执行:** 这三个组可以**同时**工作（如果硬件资源独立）。
    *   **Reward WG:** 计算 $r(x, y)$。
    *   **Critic WG:** 计算 $V(x)$。
    *   **Ref WG:** 计算 $\pi_{ref}(y|x)$。
3.  所有结果写回 Object Store。

#### 阶段 3: 优势计算 (CPU 计算)
1.  **Trainer** 从 Object Store 拉取必要的标量数据（Reward, Value, LogProb）。注意：这里不需要拉取巨大的模型权重或中间激活值。
2.  **Trainer** 在本地 CPU 上计算 GAE (Generalized Advantage Estimation) 和 Returns。

#### 阶段 4: 训练 (Update)
1.  **Trainer** 将计算好的 Advantages 和 Returns 附加到数据句柄中。
2.  **Trainer** 发出指令: `actor_wg.update_policy(batch)`。
    *   **Actor Worker** 切换回 **PyTorch FSDP 模式**（如果之前用了 vLLM）。
    *   执行标准的 PPO Loss 计算和反向传播。
3.  **Trainer** 发出指令: `critic_wg.update_value(batch)`。
    *   Critic 更新价值网络。

### 五、 最新版本的核心代码结构映射

在最新版 Verl 中，代码结构高度模块化。

1.  **配置 (Config):** 使用 Hydra 管理。
    *   定义了每个角色使用什么模型、什么并行策略（FSDP/Megatron）、多少资源。

2.  **入口 (Entry Point):**
    *   `verl/trainers/ppo_trainer.py`: 定义了 `RayPPOTrainer` 类。
    *   **关键参数 `role_worker_mapping`**: 这是一个字典，决定了每个角色使用哪个 Worker 类。这实现了算法与后端的解耦。

3.  **后端实现 (Workers):**
    *   `verl/workers/fsdp_workers.py`: 基于 PyTorch FSDP 的实现。
    *   `verl/workers/megatron_workers.py`: 基于 Megatron-LM 的实现。
    *   **模型构建 (`build_model`)**: 发生在这些 Worker 的 `__init__` 方法中。

4.  **工具库 (Utils):**
    *   `verl/utils/model.py`: 封装了 `transformers` 库，负责下载和加载 HuggingFace 模型。

### 六、 总结：Verl 的设计之美

Verl 的设计逻辑可以概括为：

1.  **控制与计算分离:** 让 Ray 负责复杂的分布式调度，让 FSDP/Megatron 负责高性能计算。
2.  **混合引擎 (Hybrid Engine):** 推理时用 vLLM (快)，训练时用 FSDP (稳)，在同一个 Worker 中无缝切换。
3.  **抽象统一:** 无论底层是单机单卡还是千卡集群，对上层的算法开发者来说，看到的都是统一的 `WorkerGroup` 接口。

这就是 Verl 最新版本的设计架构：**一个为了大规模 LLM Post-training 而生的、高度可扩展的分布式系统。**
